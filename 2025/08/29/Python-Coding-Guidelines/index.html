Python Coding Guidelines

General

  • All code must support both Python 2 and Python 3 and both POSIX and NT.
  • All code must be tested and coverage measured on both Python 2 and Python 3 and both POSIX and NT before updating to GitHub or PyPI.
  • All code must be easily portable to C++, if and when required.

Syntax

  • If using division or print(...), must from __future__ import division, print_function.
  • No async, await, yield from, walrus (:=), or match (match/case).

String and Byte Literals

  • Always use explicit string prefixes:
    • u'' for Unicode strings
    • b'' for byte strings
    • Do not use unprefixed string literals ("text").
  • All API functions accepting strings must accept typing.Text (which is unicode in Python 2 and str in Python 3).
  • Use the <project-name> package to convert typing.Text instances to properly-encoded strs. Add <project-name> to requirements.txt.

String Formatting

  • Use only % for formatting. Do not use f-strings or .format().

Typing

  • All API functions must be fully typed and use type comments only (# type: ...). No inline type annotations or AnnAssign syntax.
  • Use only typing features in Python 3.5 and PEP 484.

Classes

  • No attrs, dataclasses, namedtuples, or autogenerated class tools. Use only hand-written classes.
  • All classes must declare __slots__.
  • All classes must explicitly inherit from a base class, such as object.
  • Use __init__ for mutable, __new__ for immutable classes.
  • Use six.with_metaclass(meta, *bases) for metaclasses.

Version Checks & Imports

  • All version-dependent code must use sys.version_info, as used in typeshed.
  • Do not use except ImportError, sys.version, or other non-standard guards.

Enums

  • Do not use auto() (not in enum34).

Input Handling

  • Any code handling user input (e.g., using raw_input() in Python 2 or input() in Python 3) must import readline.

API Visibility

  • All functions and classes must be public; private APIs (names starting with _) are not permitted.
    • This includes all methods and helpers - no "internal-use only" or "hidden" interfaces.
    • All APIs must be documented and have full type comments, as per the Typing section.

Python Packaging

  • Strict correspondence required between package and module name.
    • For example, if package name is my-tool, module name must be my_tool.
  • Must include docstrings for each class and function, and a README.md for the whole package.
    • Auto-generate them using LLMs, then manually edit them.
  • Must include a LICENSE file and copyright at the top of each Python file, in this format: # Copyright (c) 2025 Jifeng Wu\n# Licensed under the X License. See LICENSE file in the project root for full license information.
    • MIT/BSD licenses for simple infrastructure (e.g., helper function/class).
    • Apache licenses for complex infrastructure (e.g., compiler framework).
    • AGPL licenses for applications (e.g., video player).
  • Must include a requirements.txt file containing at least the following:
    • typing; python_version < '3.5'
    • enum34; python_version < '3.4' (if using enums).
    • pyreadline (if handling user input).
    • six (if using metaclasses)
  • Must use pyproject.toml as sole build system/configuration file (see below). Do not use setup.py for packaging, installing, or building.
  • Must include a setup.cfg in project root with the following content to declare a universal (py2+py3) wheel (see below).

All of the rules and practices described here summarize years of real-world experience, trial, and error, and are battle tested in production - across legacy systems, multiple interpreters, and the most challenging platforms (including Windows XP and Jython2).

They are not hypothetical, but distilled from lessons hard-won at the interface of compatibility, reliability, and maintainability. Following them means your code will run anywhere Python can run, and will stand the test of time.
