Python Coding Guidelines

General

  • All code must support both Python 2 and Python 3 and both POSIX and NT.
  • All code must be tested and coverage measured on both Python 2 and Python 3 and both POSIX and NT before updating to GitHub or PyPI.
  • All code must be easily portable to C++, if and when required.

Syntax

  • If using division or print(...), must from __future__ import division, print_function.
  • No async, await, yield from, walrus (:=), or match (match/case).

String and Byte Literals

  • Always use explicit string prefixes:
    • u'' for Unicode strings
    • b'' for byte strings
    • Do not use unprefixed string literals ("text").
  • All API functions accepting strings must accept typing.Text (which is unicode in Python 2 and str in Python 3).
  • Use the textcompat package to convert typing.Text instances to properly-encoded strs. Add textcompat to requirements.txt.

String Formatting

  • Do not use f-strings.
  • Do not use .format() for strings.
  • Use only % formatting.

Typing

  • All API functions must be fully typed and use type comments only (# type: ...).
  • Add typing; python_version < '3.5' to requirements.txt.
  • No inline type annotations or AnnAssign syntax.
  • Use only typing features in Python 3.5 and PEP 484.

Classes

  • No attrs, dataclasses, namedtuples, or autogenerated class tools. Use only hand-written classes.
  • All classes must declare __slots__.
  • All classes must explicitly inherit from a base class, such as object.
  • Use __init__ for mutable, __new__ for immutable classes.
  • Use six.with_metaclass(meta, *bases) for metaclasses.

Version Checks & Imports

  • All version-dependent code must use sys.version_info, as used in typeshed.
  • Do not use except ImportError, sys.version, or other non-standard guards.

Enums

  • Add enum34; python_version < '3.4' to requirements.txt.
  • Do not use auto() (not in enum34).

Input Handling

  • Any code using raw_input() (Python 2) or input() (Python 3) must:
    • import readline.
    • Add pyreadline to requirements.txt.

Python Packaging

  • Must use pyproject.toml as sole build system/configuration file.
    • Do not use setup.py for packaging, installing, or building.

All of the rules and practices described here summarize years of real-world experience, trial, and error, and are battle tested in production - across legacy systems, multiple interpreters, and the most challenging platforms (including Windows XP and Jython2).

They are not hypothetical, but distilled from lessons hard-won at the interface of compatibility, reliability, and maintainability. Following them means your code will run anywhere Python can run, and will stand the test of time.
